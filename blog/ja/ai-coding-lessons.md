# AIにコードを1年間手伝わせて学んだこと——最大の教訓はプロンプトではなかった

この1年間、AIを活用して実際のプロダクトを構築・運用してきました。

小さなデモではありません。週末の練習プロジェクトでもありません。見た目だけ整っていれば良いようなページでもありません。

本物のプロダクトです——約3万行のコードを持つAIカスタマーサービスSaaSで、データベース、ナレッジベース、キャッシュ、ベクターデータベース、そしてRAGフローを備えています。

私は従来の意味でのエンジニアではありません。どちらかといえばプロダクト寄りの人間です。関心があるのはシンプルな問いです：機能は動くか、ユーザーは理解できるか、プロダクトは壊れないか、自分一人でメンテナンスできるか。

1年間を経て、重要なことを学びました。

**AIはエンジニアでない人がソフトウェアを作る上で本当に役立ちます。しかし難しいのはAIにコードを書かせることではなく、AIが書いたコードを本当に信頼できるかを判断することです。**

---

## ほとんどの人はプロンプトに注目する

AIコーディングの話題になると、たいていプロンプト——AIへの指示——の話になります。

「シニアエンジニアとして振る舞え。」「ステップごとに考えろ。」「クリーンなコードを書け。」

プロンプトは確かに役立ちます。しかしそれだけでは不十分です。

プロンプトは「良い仕事をしてください」と伝えるようなものです。聞こえは良いですが、実際のプロダクトはそれだけでは作れません。実際のプロダクトにはルールが必要です。

レストランを想像してください。料理人に「美味しい料理を作ってください」と言うだけでは足りません。食材の保管場所、清掃すべき箇所、触れてはいけないもの、提供前の確認方法——こうしたルールが必要です。

大きな会社では多くの人がプロダクトの品質を守ります。一人がコードを書き、別の人がレビューし、テスターが動作確認をし、自動テストが走ります。

しかし一人で作っている場合、そのような人たちは存在しません。

だから私は自分自身の安全システムを作る必要がありました。より良いプロンプトではなく、機能するプロセスを。

---

## AIはランダムに失敗するのではない

最も重要な気づき：**AIはランダムに失敗するのではなく、同じパターンで繰り返し失敗します。**

- 実際のファイルを読む前に推測する。
- 一箇所を修正して別の箇所を忘れる。
- 「完了しました」と言うが、実際のファイルは変わっていない。
- 新機能に集中して古いルールを忘れる。
- 長いファイルを編集して末尾を切り落とす。

最後のものが最も危険です。私はこれを**トランケーション（切り捨て）**と呼んでいます——ファイルが終わるべき場所の前に切り取られてしまう現象です。

プロダクトは起動するかもしれません。ページは正常に見えるかもしれません。主要な機能は動くかもしれません。問題ないと思ってしまいます。しかしファイルの末尾近くのどこかで、何かが欠けています——関数かもしれない、重要な終了処理かもしれない。

プロダクトはすぐに壊れません。後で壊れます。3回の更新後かもしれない、5回後かもしれない。ある日ユーザーが何かをクリックして、プロダクトが失敗します。

本当の問題はここから始まります——なぜなら、その失敗は今日起きたのではなく、何バージョンも前に埋め込まれたものだからです。データベースのせいかもしれない、サーバーのせいかもしれない、パッケージ更新のせいかもしれない——しかし本当の原因は、以前に静かに切り捨てられたファイルでした。

**AIが犯す最も危険なミスは、すぐに目に見えるものとは限りません。今日は問題なく見えて、後になって壊れるものこそが最も危険なミスである場合があります。**

## 私のAIコーディングルール

多くの失敗を経て、シンプルなシステムを作りました。私はこれを **D / A / B / GO / S / P** と呼んでいます。

魔法の言葉ではありません。AIにいつ読むか、いつ話すか、いつ編集するか、いつ確認するか、いつ保存するかを伝えるためのルールです。

### D — 実際のプロジェクトをダウンロード

実際のプロジェクトをサンドボックス——本番環境に影響を与えない安全なテスト環境——にダウンロードします。

AIは推測が好きです。「私のプロダクトにはナレッジベースとベクターデータベースがある」と伝えると、一般的な構成を想像するかもしれません。しかし私のプロダクトはアイデアではなく、実際のファイルを持つ実際のプロダクトです。

**推測するな。実際のファイルを見ろ。**

### A — まず確認する

AIが何かを変更する前に、理解していることを説明させます：問題は何か、どのファイルが影響を受けるか、不明な点は何か、何をするつもりか。

多くの初心者がAIに助けを求めると、AIはすぐに編集を始めます。速く感じますが、問題を誤解していれば速く間違いを犯します。

**議論なしに編集なし。**

### B — 変更リストを整理する

AIが何かを編集する前に、変更するファイル、各ファイルでの変更内容、触れないもの、リスクをリスト化させます。

そして私が確認します。

一つの小さなボタンを修正するよう頼んだのに、ページ全体を書き直すことがあります。一文を変えるよう頼んだのに、フロー全体を変えることがあります。AIは怠け者ではありません——時に積極的すぎます。

**確認済みリストなしに作業なし。**

### GO — 開始

AIは編集を開始できます——ただし確認済みの範囲内でのみ。作業中に新しい問題を見つけた場合は停止して確認を求めます。静かに他のものを変更することはできません。

遅いのは議論ではありません。本来起きるべきではなかった変更の後片付けこそが遅いのです。

### S — セルフチェック

Sは最も重要なステップです。しかし「全部確認しました。問題ないようです」というセルフチェックではありません。それはほぼ無意味です。

本当のセルフチェックは、AIがよく間違える点を確認します：

- 実際のファイルは本当に変更されたか？
- 大きなファイルが静かに保存されなかったか？
- ファイルの末尾が切り取られていないか？
- ファイルの行数は妥当か？
- 古い変数名が残っていないか？
- 表示テキストがHTMLに直書きされていないか？
- 翻訳ファイルはすべての有効言語で更新されたか？
- 権限、キャッシュ、データ同期は見落とされていないか？

### P — クリーンなコミットをプッシュ

Gitへのプッシュは、一つのタスクに対して一つのクリーンなコミットとします。ファイルごとに一つずつプッシュすると履歴が乱雑になります。

後で何か壊れたとき、どの変更が原因かを知る必要があります。クリーンな履歴がデバッグを可能にします。

## 危うく騙されそうになったミス

あるとき、セルフチェックリストに二つの項目を追加すべきかAIと議論していました——async/awaitの処理に関するものと、キャッシュとembeddingの更新に関するものです。

AIはこれらのチェックが重要だと言いました。私のプロダクトにはasyncコード、キャッシュ、ベクターデータベース、ナレッジベースがあるので注意すべきだと。とても賢そうに聞こえました。あまりにも合理的に聞こえたので、自分を疑いそうになりました。

そこでAIに推測をやめて実際のコードを読むよう指示しました。

結果は明確でした。それらの問題はすでに対処済みでした。キャッシュは適切な場所でクリアされており、embeddingもデータとともに更新されていました。なぜ一部のキャッシュをクリアすべきでないかを説明するコメントまでコードの中にありました。

AIは隠れた問題を見つけたのではありません。**実際のファイルを読む前に疑念を生み出したのです。**

これがAIの最も危険な側面の一つです。間違ったコードを書くだけではなく、自信を持った口調で、すでに正しいものを疑わせることもあるのです。

**AIに推測させるな。実際のファイルを確認させろ。**

## プロトコルの要約版

ご自身のプロジェクト用にコピーして調整してください。

```
開始：
- プロジェクトをサンドボックスにダウンロードする。
- すべての変更はサンドボックス内で行い、本番環境に直接変更しない。

基本ルール：
- 明確に指示した部分のみを変更する。
- 改善したい場合は説明して確認を待つ。
- 不明な点があれば先に質問する。推測しない。
- ファイルが切り捨てられている場合、Gitの元バージョンと比較してから修正する。
- パッケージを追加する前に理由を説明して確認を待つ。

i18nルール：
- すべての表示テキストは翻訳システムを使用する。
- 表示テキストをHTMLに直書きしない。
- すべての有効な言語の翻訳キーを追加する。

ツールルール：
- 新規ファイル → 作成ツールを使用。
- 小さなファイル（50KB未満）→ 対象を絞った編集。
- 大きなファイル（50KB以上）→ ファイル全体を書き直さない。
- 編集後はディスク上の実際のファイルを確認する。ツールの報告だけを信じない。

プロトコル：
D — 実際のプロジェクトをサンドボックスにダウンロード。
A — ファイルに触れる前に確認と議論。
B — 変更リストを整理して確認を待つ。
GO — 確認後のみ開始。
S — 実際のファイル、末尾の完整性、行数、差分、古い参照、i18n、権限、キャッシュ、データ同期をセルフチェック。
P — チェック通過後にクリーンなコミットをプッシュ。
```

## 初心者へのメッセージ

AIは誰がソフトウェアを作れるかを変えつつあります。以前はコードが書けなければ本格的なソフトウェアプロダクトを作るのは難しかった。今は始めることができます。

しかしリスクは消えていません。形が変わっただけです。

以前の最大の問題：「コードが書けない。」

今の最大の問題：「AIがコードを書いた。でも安全かどうかわからない。」

**「完了しました」は「信頼できる」を意味しません。**

- AIが「完了」と言っても、実際のファイルが変更されたとは限りません。
- ページが開いても、システムが完整とは限りません。
- 機能が一度動いても、本番リリースして良いとは限りません。

完璧なプロンプトを追いかけるのではなく、自分自身のルールを作りましょう。AIがミスをするたびに書き留める。同じミスが再発したらチェックポイントにする。やがてチェックポイントが安全システムになります。

短期的には、より良いプロンプトを持つ人の方が速く動けるかもしれません。長期的には、より良いプロセスを持つ人がより信頼性の高いプロダクトを作ります。

本当の差は、AIを使う人と使わない人の間にあるのではありません。AIに作らせるだけの人と、AIが作ったものを本当に信頼できるか確認する方法を知っている人の間にあります。
